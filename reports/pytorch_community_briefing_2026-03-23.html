CRITICAL NPU #178123 - Fix float32 overflow in variance computation for normalization layers

Pull Request · 作者: gaurav-redhat · 日期: 2026-03-23 · 标签: module: cpu, release notes: nn

问题：BatchNorm、LayerNorm、GroupNorm、InstanceNorm 在 float32 大数值输入时（约 1e30），Welford 在线算法的方差计算会溢出为 inf，导致输出 NaN。

解决方案：对 float32 输入使用 float64 累加器进行 Welford 计算，保持向后兼容。

// 核心修复：累加器精度提升 template <typename T> using acc_type = typename std::conditional< std::is_same<T, float>::value, double, // Promote float32 to float64 T // Keep other types as-is >::type;

影响文件：batch_norm_kernel.cpp, layer_norm_kernel.cpp, group_norm_kernel.cpp, Normalization.cuh

[P0] NPU 团队行动：检查 NPU normalization 实现是否使用累加器提升；在 NPU 测试套件中添加大数值输入测试。
信息来源: PR diff, 代码分析

CRITICAL NPU #178020 - Enable initializing embedded constants on privateuse1 backend

Pull Request · 作者: arfaian · 日期: 2026-03-20 · 标签: open source

问题：FakeTensor.avoid_device_init 机制不尊重 PrivateUse1 后端，导致 embedded constants 被错误地放在 CPU 而非目标设备，使 torch.compile 在 NPU 上无法工作。

// 修复前：总是回退到 CPU if self.avoid_device_init: return tensor.cpu() // 修复后：检查 PrivateUse1 后端注册 if self.avoid_device_init: if device.type == 'privateuse1' and torch._C._is_privateuse1_backend_registered(): return tensor.to(device) // 正确行为 return tensor.cpu()

关键影响：这是 NPU 支持 torch.compile 的必要条件，直接影响 NPU 后端的编译功能可用性。

[P0] NPU 团队行动：立即在 NPU 后端测试此 PR，验证 embedded constants 正确放置；运行完整 torch.compile 测试套件。
信息来源: PR diff, Issue #178019, 代码分析

CRITICAL NPU #178019 - FakeTensor.avoid_device_init not respecting privateuse1 backend

Issue · 作者: arfaian · 日期: 2026-03-20 · 标签: module: backend, module: fakeTensor, module: PrivateUse1

问题描述：在 FakeTensor 模式下创建 embedded constants 时，即使指定了自定义设备，张量也会被错误地放置在 CPU 上，导致设备不匹配错误。

// 复现代码 def simple(x): c = torch.tensor([1, 2, 3], device=x.device) return x + c x = torch.tensor([1, 2, 3], device=my_custom_device) model = torch.compile(simple, backend=my_custom_backend) out = model(x) // c 被放在 CPU 而不是 my_custom_device！

解决方案：PR #178020 已提供修复，等待合并。

[P0] NPU 团队行动：跟踪 PR #178020 合并状态；在 NPU 后端应用补丁验证修复效果；临时 workaround：使用 FakeTensorMode(avoid_device_init=False)。
信息来源: Issue body, PR #178020

HIGH #178116 - Add _dtensor::mesh_get_process_group custom op

Pull Request · 作者: aorenste · 日期: 2026-03-23 · 标签: module: inductor, release notes: distributed (dtensor)

背景：compile_on_one_rank 是 PyTorch 分布式训练的重要优化，允许在一个 rank 上编译模型，然后在所有 rank 上执行。

问题：DeviceMesh 包含 ProcessGroup 对象，这是"不透明对象"（Opaque Object），不能直接通过 FX 图流式传输。之前的实现通过字符串名称查找 ProcessGroup，导致运行时开销和跨 rank 输出代码不一致。

// 新的自定义操作 @torch.library.custom_op("_dtensor::mesh_get_process_group", mutates_args=()) def mesh_get_process_group(mesh: DeviceMesh, dim: int) -> ProcessGroup: return mesh.get_process_group(dim)

架构意义：此 PR 是 10 个 PR 堆栈的一部分，引入了 PyTorch 对"不透明对象"作为图输入的全面支持。NPU 特定的对象（如 NPUStream、NPUEvent）可以类似地作为不透明对象支持。

[P1] NPU 团队行动：了解 OpaqueObjectState IR 节点设计；评估 NPU 自定义对象是否需要类似支持；测试 DTensor + torch.compile 在 NPU 上的兼容性。
信息来源: PR diff, 堆栈上下文分析

HIGH #178016 - Fix eager/compiled mismatch for integer floor_divide with zero divisor

Pull Request · 作者: dcci · 日期: 2026-03-20 · 标签: module: inductor, release notes: inductor

问题：CUDA 上整数 floor_divide 在除数为零时，eager 模式返回 0，compiled 模式返回 -1，存在静默正确性差异。

// 修复策略：统一两种路径的零除数行为 // Eager 路径 (c10::div_floor_integer) if (b == 0) return 0; // Compiled 路径 (Triton) tl.where(b == 0, 0, tl.floordiv(a, b))

修复范围：CPU kernel (BinaryOpsKernel.cpp) 和 Triton codegen (triton.py) 均添加零除数检查。

[P1] NPU 团队行动：检查 NPU 后端整数 floor_divide 实现；验证零除数行为是否与 PyTorch 一致（应返回 0）；添加测试用例覆盖零除数场景。
信息来源: PR diff, Issue #178013

HIGH NPU #178023 - [AMD] Fix num_warps when max_autotune is enabled on HIP

Pull Request · 作者: yvonne-lab · 日期: 2026-03-20 · 标签: module: inductor, ciflow/inductor

问题：AMD MI350 上启用 max_autotune 时，num_warps=32 导致 threads_per_workgroup=2048（32×64），超过硬件限制 1024，引发启动失败。

// 修复：平台特定的 num_warps 限制 def get_max_num_warps_for_platform(): if torch.version.hip: // AMD/HIP return 16 // 16 * 64 = 1024 else: // NVIDIA/CUDA return 32 // 32 * 32 = 1024

NPU 参考价值：NPU 通常也有特定的 warp/wavefront 大小和线程限制。此修复提供了平台特定配置限制的参考模式。

[P1] NPU 团队行动：分析 NPU 线程组织模型（warp 大小、最大线程数）；审查当前 NPU 自动调优配置是否超出硬件限制；实现 NPU 特定的 num_warps 限制。
信息来源: PR diff, 硬件规格分析

MEDIUM #178066 - [RFC] All-to-all permute for Ulysses Parallel

// 提议的新 API def all_to_all_permute( input: torch.Tensor, group: ProcessGroup, scatter_dim: int, // 通常是序列维度 gather_dim: int, // 通常是 head 维度 ) -> torch.Tensor:

优势：比 Ring Attention 通信更少（2 次 all-to-all vs P-1 次点对点），与 Tensor Parallel 正交可组合。
