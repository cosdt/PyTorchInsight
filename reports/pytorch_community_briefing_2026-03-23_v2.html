极高 [RFC] All-to-all permute for Ulysses Parallel

RFC / Feature Request · kwen2501 · 2026-03-21 · Labels: oncall: distributed

Ulysses Parallel (UP) 是一种针对Transformer注意力模块的优化技术，用于处理长序列训练。核心机制需要通过高效的all-to-all通信实现序列维度和head维度的转换。

技术细节：当前每个rank持有 1/P 的序列和所有head状态，需要转换为持有完整序列但只持有 1/P 的head状态。现有的all-to-all操作可能不够高效，需要针对Ulysses场景优化permute操作。

# 伪代码示意Ulysses Parallel的通信模式 # 当前：每个rank有 (seq_len/P, num_heads, head_dim) # 目标：每个rank有 (seq_len, num_heads/P, head_dim) # 需要：高效的all-to-all permute
NPU适配影响：高 - 需要NPU后端支持高效的all-to-all collective操作，通信模式与计算重叠优化是关键。
建议行动: 评估NPU all-to-all性能，准备Ulysses支持计划
信息来源: GitHub Issue #178066, Meta Distributed团队

极高 [RFC] Adding Kernel-Level Profiling Support for PrivateUse1 Backends

RFC / Feature Request · vishalgoyal316 · 2026-03-20 · Labels: oncall: profiler

PrivateUse1后端目前缺乏内核级性能分析能力，无法使用Kineto进行细粒度性能剖析。此RFC提案引入PrivateUse1TracingApi C API，支持动态插件加载，无需修改Kineto即可集成。

// 预期的C API设计 // PrivateUse1TracingApi - 类似CUDA CUPTI的接口 void PrivateUse1TracingApi_init(); void PrivateUse1TracingApi_startTracing(); void PrivateUse1TracingApi_stopTracing();

关键价值：这是NPU适配的基础设施级功能，没有profiling支持，性能优化无从谈起。华为昇腾、寒武纪等NPU厂商急需此功能。

NPU适配影响：极高 - 基础设施级功能，必须支持。建议6个月内完成Kineto插件实现。
建议行动: 评估Kineto插件实现，准备NPU profiling方案
信息来源: GitHub Issue #177978, NPU厂商代表

极高 [RFC] Add A Cross-Backend Device Name API

RFC / Feature Request · pkourdis · 2026-03-20 · Labels: feature, module: accelerator

当前获取设备名称需要后端特定代码（CUDA: torch.cuda.get_device_name()，XPU: torch.xpu.get_device_name()），缺乏统一的跨后端API。

# 当前用法（后端特定） if torch.cuda.is_available(): name = torch.cuda.get_device_name(0) elif torch.xpu.is_available(): name = torch.xpu.get_device_name(0) # RFC提案（统一接口） name = torch.accelerator.current_device_name() # 后端无关

设计原则：与现有torch.accelerator命名空间保持一致，支持CUDA、XPU、PrivateUse1等所有后端。

NPU适配影响：极高 - 基础设施级API，所有NPU厂商自动受益。PrivateUse1后端自动获得设备名称支持。
建议行动: 确保NPU设备名称API正确实现，提供实现反馈
信息来源: GitHub Issue #177935, Intel/Qualcomm贡献者

高 dist.new_group(..., use_local_synchronization=True) hangs with Gloo

Bug Report · tie-pilot-qxw · 2026-03-20 · Labels: oncall: distributed, module: c10d

使用torch.distributed.new_group(..., use_local_synchronization=True)创建重叠process group时，Gloo后端发生hang，而NCCL后端在相同代码下成功。

# 创建多个重叠process group for group_ranks in overlapping_groups: group = dist.new_group( group_ranks, use_local_synchronization=True # 导致Gloo hang )

关键观察：每个rank以一致的顺序调用new_group()，文档警告use_local_synchronization=True可能在创建多个重叠组时死锁，但NCCL成功而Gloo失败，表明后端行为不一致。

NPU适配影响：中 - 如果NPU使用类似Gloo的通信机制，可能同样受影响。需要验证NPU backend在重叠组场景下的行为。
建议行动: NPU通常使用NCCL，影响较小，但需验证复杂group拓扑稳定性

高 FakeTensor.avoid_device_init not respecting privateuse1 backend

Bug Report · arfaian · 2026-03-20 · Labels: triaged, module: backend, module: PrivateUse1

使用torch.compile编译自定义后端（PrivateUse1）的函数时，内部常量张量被错误地放置在CPU上，即使指定了自定义设备，设备初始化未正确传播。

def simple(x): c = torch.tensor([1, 2, 3, 4, 5], device=x.device) return x + c x = torch.tensor([1, 2, 3, 4, 5], device=my_custom_device) model = torch.compile(simple, backend=my_custom_backend) out = model(x) # c被放在CPU而不是my_custom_device

根本原因：FakeTensor.avoid_device_init机制未正确处理PrivateUse1后端，在aot_autograd后，常量张量的设备信息丢失。

NPU适配影响：极高 - 这是NPU使用torch.compile的关键障碍。torch.compile是PyTorch 2.x的核心特性，NPU必须支持才能提供竞争力。
建议行动: 高优先级推动修复，测试NPU + torch.compile兼容性
信息来源: GitHub Issue #178019, NPU厂商报告

高 torch.compile streams support in AOTI

Feature Request · mlazos · 2026-03-23 · Labels: feature, module: inductor, module: compiled streams

AOTI (Ahead-of-Time Inductor) 当前streams实现依赖Python对象查找表，需要支持C++级别的streams操作。提案使用不透明对象(opaque objects)表示streams，在C++层实现streams管理。

// 期望的解决方案 // 方案1: Opaque Stream Handle struct OpaqueStreamHandle { void* native_handle; DeviceType device_type; }; // 方案2: C++ Stream Manager class AOTIStreamManager { public: void* getStream(int stream_id); void recordEvent(void* event, void* stream); void waitEvent(void* event, void* stream); };

技术挑战：Dynamo使用Python字典跟踪streams和events，AOTI生成C++代码无法直接访问Python对象，需要opaque对象或C++等效机制。

NPU适配影响：高 - AOTI是PyTorch部署的未来方向，NPU需要提供stream支持才能完整支持AOTI。需要定义NPU stream的opaque handle。
建议行动: 跟踪RFC进展，设计NPU AOTI stream方案
信息来源: GitHub Issue #178132, PyTorch Compiler团队

中 Functional Collectives shouldn't assert contiguous inputs

Enhancement · wconstab · 2026-03-19 · Labels: oncall: distributed, module: c10d

Functional collectives API在Python层断言输入是contiguous的，但底层NCCL API支持non-contiguous输入。这种断言限制了灵活性，可能导致不必要的拷贝。

# 当前代码（有限制） def all_reduce(tensor, ...): assert tensor.is_contiguous(), "Input must be contiguous" ... # 期望的代码（更灵活） # 让后端决定如何处理non-contiguous输入 # 后端可以：1. 如果支持直接使用 2. 如果不支持，内部调用contiguous()

提案：移除Python层的contiguous断言，让后端实现决定如何处理non-contiguous输入，减少不必要的tensor拷贝，提升分布式训练性能。

NPU适配影响：中 - NPU collective实现需要评估对non-contiguous的支持。如果NPU不支持，需要内部处理，定义清晰的fallback策略。
建议行动: 评估NPU collective对non-contiguous输入的支持能力
信息来源: GitHub Issue #177902, Meta Distributed团队
